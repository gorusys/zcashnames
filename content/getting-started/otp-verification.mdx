# OTP Verification

Admin actions (list, delist, update) require one-time password verification to prove the caller controls the registered address. This is powered by ZVS (Zcash Verification Service).

## How it works

1. The interface generates a verification memo containing a unique session ID and the registered address
2. The user sends this memo to the ZVS address as a shielded transaction (0.002 ZEC)
3. ZVS detects the transaction in the mempool and computes a 6-digit OTP using HMAC-SHA256
4. ZVS sends the OTP back to the user's address as a shielded memo
5. The user enters the OTP in the interface to authorize the action

## Why this exists

ZNS has no accounts or passwords. The only way to prove ownership of a name is to prove control of the Zcash address it's registered to. OTP verification does this by requiring a transaction from that address and delivering a response to it.

## Memo format

```
DO NOT MODIFY:{zvs/<session_id>,<your_address>}
```

- `session_id`: 16-digit cryptographically random number that seeds the OTP
- `your_address`: the unified address registered to the name

## Security properties

- **Stateless:** the OTP is deterministic (HMAC-SHA256 of session ID + address), so both ZVS and the web app compute the same code independently
- **Shielded:** all verification transactions use Orchard shielded notes — the OTP is never visible on-chain
- **No login:** there is no account to compromise; verification is per-action
- **Address-bound:** the OTP is derived from both the session ID and the address, preventing replay with a swapped address
